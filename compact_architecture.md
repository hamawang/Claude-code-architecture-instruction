# `compact` 模块代码逻辑与上下文防爆架构解析

`src/services/compact/` 文件夹是 Claude Code 的**上下文极限管理中心 (Context Window Management)**。
在大语言模型（LLM）的对话中，如果任由对话流无限制增长，最终不仅会遭遇 API 的硬性 Token 长度上限 (Max Context Limits)，还会导致 Prompt 响应越来越慢、价格越来越高。

为了解决这个问题，这个目录下实现了一套**金字塔式的三级防爆与阻断折叠体系**。

---

## 第一级防线：即时微观清理 (`microCompact.ts`)

在真正触及上限前，系统已经在偷偷抛弃无用负载。这个文件主要清理那些**体积大但时效性差的占位符**（比如刚打印的一大长串 `grep` 搜索结果、`cat` 取出的冗长旧代码）。
它有两种触发与裁剪模式：

1. **时间断层裁剪 (Time-based gap trigger)**：
   如果是你昨晚聊到一半挂机的会话，第二天接着聊，旧的 Prompt Cache 肯定已经在服务器端过期（通常缓存生命周期就几分钟）。一旦探测到两次对话的间隔时间 （`gapMinutes`）极长，系统会直接把前面所有大体积只读工具的返回结果替换成字符串：`[Old tool result content cleared]`。
   既然必须重新计算 Token 费，那就彻底扔掉无效行囊。

2. **API 缓存无损编辑 (Cached MC)**：
   在会话高频连贯时跑的模型，如果遇到超长输出，它会尝试使用一套极其先锋的后端缓存编辑机制（Cache Edits block）。通知 API 服务器“在不破坏此前庞大的 Prompt 前缀缓存连贯性的前提下，只从树上摘除几个旧的 Tool Result 分支”。这种方法不仅省钱还能维持几百毫秒的首屏极速响应。

---

## 第二级防线：无缝嫁接后台记忆库 (`sessionMemoryCompact.ts`)

这是由 `autoCompact.ts` 监控和调度的优先方案，也是对前面分析的后台背景子代理（Session Memory Background Agent）的**神级收网**运用。

当主对话的 Token 量距离阈值仅剩不到 `13,000` (配置常量 `AUTOCOMPACT_BUFFER_TOKENS`) 缓冲量时：
传统的对话客户端（包括旧版 Claude Code）会立即在这时候卡转圈——它须要去另外开一个沉重的模型调用，把前文完整吃下去，生成一段汇总文本。

但由于前面提过，Claude Code 一直有一个“影子工会”在后台偷偷读我们的对话并在磁盘上写 `SESSION_MEMORY.md` 短篇总结（甚至帮我们做好了章节梳理）。
因此，本模块选择**“摘果子”**：
1. 它直接读取硬盘里那份不到 2000 个 Token 的旧总结文件。
2. 调用 `calculateMessagesToKeepIndex` 函数拉网，计算出哪些对话是后台还没来得及写的“新鲜肉”。为了保证不产生幻觉崩溃，它还会仔细地对齐结构，绝对不会把模型的 `[思考锁] -> [发工具] -> [获结果]` 这一连贯三连招拦腰劈断。
3. 把前半截废老对话直接暴力删除，在界面原位凭空注射一份轻量级的带有前面摘要文件的**“系统级紧凑幽灵节点”**，随后拼接后半截的聊天。**0 额外发请求，0 等待延迟**！

---

## 第三级防线：传统高压浓缩机 (`compact.ts` - Legacy)

如果前面两步都失效（比如后台记忆提取线程被打断，一直没写成功；或者你直接用了一行命令让它读取 15 万行极其致密的新代码）。
那么最后一道底线网就会被兜醒。这就类似于核弹级别的一次性推倒重来。

它会单独发配一个最高权限的长调用给模型，利用专门的重组提示词，读入这几十万字废话，要求模型强制吐出纯净的压缩摘要结构，随后把当前所有的上下文节点清空抹平为 0。

---

## 4. 重点启示：工程师思维的架构防御

你看，当绝大部分的 AI 衍生客户端还在用类似 `messages.splice(0, length - 10)` 这种粗暴野蛮、且容易导致上下文语境断裂丢失的切片方法来防止提示词爆盘时。
Claude 团队构建了一个立体的、防缓存穿透的阶梯网。优先切掉旁枝末角（microCompact），再尝试复用廉价的备胎成果（SessionMemoryCompact），最后才动用昂贵的终极重构（Legacy Compact）。这才是打造真正 Agent “耐力赛引擎”应当具备的素质！
